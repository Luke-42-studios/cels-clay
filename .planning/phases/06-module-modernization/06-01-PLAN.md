<objective>
Modernize cels-clay to CELS v0.4+ patterns and remove all deprecated Feature/Provider code.

Purpose: The current codebase uses `cel_module()` (deprecated), `atexit()` for cleanup, and carries dead Feature/Provider infrastructure (ClayRenderable_feature_id, _clay_feature_counter, ClayRenderable_register). This plan replaces these with CEL_Module, CEL_Lifecycle, CEL_Define_State, and removes all Feature/Provider remnants -- establishing a clean foundation for Phases 7-11.

Output: Modernized clay_engine.h/c using CEL_Module(Clay_Engine, init) with lifecycle observers for arena management, ClayEngineState singleton with cross-TU registry, cleaned clay_render.h/c with no Feature/Provider code, and ClaySurface with corrected lifecycle type.
</objective>

Modernize the Clay engine module to CELS v0.4+ patterns, following the cels-sdl3 reference.

**clay_engine.h changes:**

1. Replace the current `extern cels_entity_t Clay_Engine;` + `extern void Clay_Engine_init(void);` with:
   ```c
   CEL_Module(Clay_Engine);
   ```

2. Add state singleton definition:
   ```c
   CEL_Define_State(ClayEngineState) {
       bool initialized;
   };
   ```

3. Remove `Clay_Engine_use()` declaration -- module registration replaces it. Configuration is done via a static configure function:
   ```c
   extern void Clay_Engine_configure(const ClayEngineConfig* config);
   ```
   This stores config before module init (same pattern as Clay_NCurses_configure in Phase 9).

4. Remove `clay_layout_use()` and `clay_render_use()` declarations (advanced sub-module API is retired).

5. Remove `#include "cels-clay/clay_render.h"` from clay_engine.h -- the engine header should not depend on the render header.

**clay_engine.c changes:**

1. Replace `cel_module(Clay_Engine) { ... }` with:
   ```c
   CEL_State(ClayEngineState);

   CEL_Lifecycle(ClayEngineLC);

   CEL_Observe(ClayEngineLC, on_create) {
       // Read stored config from g_clay_config
       // Initialize Clay arena (same code as current cel_module body steps 1-4)
       // Register state with cross-TU pointer registry:
       //   ClayEngineState_register();
       //   cels_state_bind(ClayEngineState);
       // Set ClayEngineState.initialized = true
   }

   CEL_Observe(ClayEngineLC, on_destroy) {
       // Free frame arena via _cel_clay_layout_cleanup()
       // Free Clay arena memory (g_clay_arena_memory)
       // Set ClayEngineState.initialized = false
   }

   CEL_Module(Clay_Engine, init) {
       cels_register(ClayEngineState, ClayEngineLC);
       cels_register(ClaySurfaceConfig, ClayUI);
       cels_register(ClayRenderableData);
       // Register layout and render systems
       _cel_clay_layout_init();
       _cel_clay_render_init();
       _cel_clay_layout_system_register();
       _cel_clay_render_system_register();
   }
   ```

2. Replace `Clay_Engine_use()` with `Clay_Engine_configure()`:
   ```c
   void Clay_Engine_configure(const ClayEngineConfig* config) {
       if (config) g_clay_config = *config;
   }
   ```
   Developers call this before cels_register(Clay_Engine) in their CEL_Build block.

3. Remove `atexit(clay_cleanup)` -- lifecycle on_destroy handles cleanup.

4. Remove `clay_layout_use()` and `clay_render_use()` functions.

5. The Clay arena initialization code (SetMaxElementCount, MinMemorySize, malloc, CreateArena, Initialize) moves from the cel_module body into CEL_Observe(ClayEngineLC, on_create). The lifecycle observer triggers when a ClayEngineConfig component is added to an entity.

   Wait -- actually the lifecycle needs to be bound to an entity. Looking at the cels-sdl3 pattern: the SDL3Context composition calls `cels_lifecycle_bind_entity(SDL3_ContextLC_id, cels_get_current_entity())`. For Clay_Engine, the lifecycle binds to the engine entity created during module init. The on_create observer runs the arena initialization.

   Revised approach: Keep arena init in the CEL_Module body (like cels-sdl3 does its registration there). The lifecycle handles cleanup on entity destruction. Actually, looking more carefully at sdl3_module.c, the init happens in the observer (sdl3_init called from on_create). We need an entity with a config component to trigger the lifecycle.

   Simplest approach: The CEL_Module body creates a "ClayEngine" entity with a ClayEngineConfig component, which triggers the ClayEngineLC on_create observer to do arena init. This mirrors SDL3Context creating an entity with SDL3_ContextConfig.

   But we can also just do the arena init directly in the module body (it runs once at registration) and use the lifecycle only for cleanup. Let's keep it simple:
   - CEL_Module body: arena init, state registration, subsystem init
   - CEL_Lifecycle on_destroy: cleanup
   - This is simpler than SDL3's pattern since Clay_Engine doesn't need per-entity lifecycle

   Actually, let's follow the SDL3 pattern exactly for consistency:
   - CEL_Module body: registers components, state, lifecycle, systems
   - A composition or manual entity creation triggers on_create for the actual init
   - ClaySurface or a new ClayEngineContext composition triggers the lifecycle

   For simplicity in v0.6, let's do: the CEL_Module body does the init directly (arena allocation, state bind) since there's only one Clay instance. Use the lifecycle for cleanup only. This avoids needing a separate trigger entity.
